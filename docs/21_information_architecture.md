# 21. Information Architecture v2 — 「毎日開きたくなる犬の健康アプリ」

対象コミット時点の実装に対する情報設計の全面見直し。**docsだけでなくアプリ本体に実装済み**。
指針: Apple / Oura / Gentler Streak — 「少ない画面で多くのことができる」。
機能は減らさず、**ユーザーが普段見る画面を3つに絞る**。

## 1. Information Architecture

```
HydroPaw
├── 🏠 Home ……「今日、うちの犬は元気?」への答えと、測定の入口
│   ├── 見守りリング(状態色) = 測定開始ボタンでもある
│   ├── 状態の言葉 → 変化 → 行動 → 最終測定 (数値なし)
│   ├── ここ7日のようす(無数値ライン)
│   ├── ─ 履歴を見る → 日誌 (Home配下 /home/history)
│   │    ├── 日誌タイムライン(測定+健康日誌、今日/きのう/おととい…)
│   │    ├── カレンダー表示(必要な時だけ切替)
│   │    └── 測定詳細 (/home/history/detail) … ppm/品質/信頼度/温湿度/グラフ
│   └── ─ きょうの記録 → 日誌入力シート(モーダル)
│   └── [フルスクリーン] 測定中 → 解析 → 結果 (/measure/session → /measure/result)
├── 🐶 Dogs …… 多頭飼いのカードスワイプ切替 + プロフィール編集/追加 (/dogs, /dogs/edit)
└── ⚙️ Settings …… デバイス/接続/電池/デモ/言語/ログアウト (/settings, /connect)
```

Bottom Navigationは **Home / Dogs / Settings の3つのみ**(実装: `app_shell.dart`)。

## 3. ユーザーフロー(主要3本)

**F1 毎日の測定(最短2タップ)**: アプリを開く → 状態が見える(0タップ) → リングorCTA →
(未接続なら接続画面を挟む) → 測定中(自動進行: WARMUP→呼気→解析) → 結果 → 自動保存 → ホーム。

**F2 日々の記録(3タップ)**: ホーム「きょうの記録」→ 種別チップ(+様子3段階) → 保存。
散歩/食欲/排便/薬/体調/メモが構造化データ+自由記述で保存される。

**F3 振り返り**: ホーム「履歴を見る」→ 日誌(今日/きのう/おととい…に測定と記録が混在) →
気になる測定をタップ → 詳細(数値・グラフ)。月をまたぐ確認だけカレンダーに切替。

## 4. 各画面の役割

| 画面 | 役割 | 住んでいい情報 | 住んではいけない情報 |
|---|---|---|---|
| Home | 安心感と測定の入口 | 状態の言葉・行動・無数値ライン | ppm・品質・接続の技術情報 |
| 測定中/結果 | 儀式としての測定体験 | 実況の言葉・品質の言葉化 | 生波形の数値羅列 |
| 日誌 | 暮らしの記録の置き場 | 測定の状態語+日誌行・時刻 | 詳細数値(→詳細へ) |
| 測定詳細 | 技術情報の唯一の住所 | ppm/Quality/Confidence/温湿度/グラフ | — |
| Dogs | 家族(犬)の切替と管理 | 写真・基本情報・見守り中表示 | 測定データの詳細 |
| Settings | 技術・アカウントの隔離先 | 接続/電池/デモ/言語/ログアウト | 健康情報 |

## 6. 健康日誌の設計 (実装: features/records/)

- **記録できるもの**: 散歩 / 食欲 / 排便 / 薬 / 体調 / 自由メモ (`CareNoteType`)
- **入力は3タップ**: 種別チップ → 様子(良い/ふつう/気になる ※食欲·排便·体調のみ) → 保存。メモは任意。
- **構造化+自由記述**: `type`(enum) と `rating`(3段階) がAI解析用の構造化データ、
  `memo` が飼い主の言葉。`schema: 1` フィールドで将来のスキーマ進化に備える。
- **保存先**: `users/{uid}/dogs/{dogId}/careNotes/{id}` (firestore.rules追加済み、
  オフライン/デモは `InMemoryCareNoteRepository`)。
- **表示**: 日誌タイムラインに測定と時系列で混在。「気になる」は琥珀色で静かに目立たせる。

## 7. カレンダーの扱い

普段は「今日・きのう・おととい・日付」の日誌形式(スクロール)。
月表示は日誌画面右上のアイコンで**必要な時だけ**切替(実装: `_CalendarView`)。
記録のある日に状態色の点を打ち、タップでその日の記録を下に表示。月表示は主役にしない。

## 8. 多頭飼いUX

- 切替は**アカウント切替ではなくカードの左右スワイプ**(Dogsタブ, `PageView`)。
- スワイプした瞬間に `selectedDogIdProvider` が更新され、ホームのリング・日誌・測定が
  すべてその犬に切り替わる(触覚フィードバックつき)。
- 末尾ページは常に「＋新しい犬を追加」。犬0頭でも同じ導線で迷わない。
- 将来: カードに犬ごとの状態色リングを出す(現在は見守り中の犬のみ評価表示)。

## 9. 将来拡張と設計理由

| 将来機能 | どこに収まるか | 破綻しない理由 |
|---|---|---|
| AI解析 | Homeの「状態の言葉」を置換/日誌に洞察行を追加 | 意味層(insights)がUIと分離済み。日誌は構造化データで学習可能 |
| 獣医共有 | 測定詳細・日誌に「共有」アクション | 詳細=技術情報の唯一の住所なので共有物が明確 |
| 複数デバイス | Settings > 測定デバイスが一覧化 | デバイスはSettingsに隔離済み、タブは増えない |
| 家族共有 | Dogsの犬カードに「家族」欄 | 犬中心のIA(uid→dogs)なので共有単位=犬で自然 |

タブを増やさず、すべて既存3タブの**深さ方向**に収まる。

## 10. UXレビュー (Apple HI / Oura / Withings視点)

**削った画面**: 測定タブ(開始前画面) — 「開始ボタンのためだけの画面」はOuraなら作らない。
ホームのリング=開始ボタンに統合した。履歴タブ — 毎日見るものではなく、Homeの深さ方向へ。

**統合した画面**: ホーム+測定開始 / 履歴+日誌+カレンダー / 犬プロフィール+犬切替。

**ユーザーが迷う点として残るもの(今後の宿題)**:
1. 初回接続フロー — CTA→接続画面→自動popは良いが、初めての人に「なぜ接続が要るか」の
   1枚(オンボーディング)がない。
2. 日誌の時刻は常に「いま」— 後から「朝の散歩」を記録できない(時刻編集は未実装)。
3. カレンダーの点は読み込み済み範囲(測定20件+ページング)に依存し、過去月は不完全な
   ことがある。月単位クエリは将来課題。
4. Dogsカードの評価表示は「見守り中の犬」のみ。全カード同時表示は評価Providerの
   family化が必要(将来)。

# v2.1 追補 — 犬のライフサイクルと頭数管理 (Flutter / Web 両実装済み)

- **Home**: 1段目=名前入り主CTA「◯◯の測定をはじめる」/2段目=履歴・記録。
  見守り中の犬をリングの左右スワイプで切替(名前+ドット+1/N表示)。
  測定開始前に対象犬の確認シート(写真+名前)を挟み、開始時点で対象を固定。
  見守り中0頭は空状態+登録導線。
- **犬の3状態**: A未設定(名前なし=保存時にのみ正式作成、残骸は削除可) /
  B設定済み・記録なし(警告つき完全削除可) / C記録あり(完全削除不可、
  「見守りを終了」= archivedフラグ。データ保持・再開可)。
- **Dogs**: 横スワイプ廃止→同カードデザインの縦一覧。編集ボタンの下に十分な
  余白を空けて控えめな削除/終了/再開テキスト。終了犬は専用見出し配下。
- **頭数上限(maxDogs)**: 初回に1問だけ質問(§9)。Settingsの
  「愛犬の登録設定」(デバイスとデータの間)で変更。上限は見守り中のみを数える。
  超過時は理由+「設定を開く」。減数時は終了する犬を選択させる(記録は保持)。
- **整合性**: 記録のdogId紐付けは不変。選択犬の消失時は先頭の見守り犬へ
  フォールバック、0頭なら空状態。状態はFirestore/localStorageに永続化。

# v2.2 追補 — ホーム全面スワイプ / 愛犬写真 / 文言・選択UXの磨き込み (両実装済み)

- **Home §1**: 主CTAを大型(min 58px)に復元。1段目=「◯◯の測定をはじめる」全幅、
  2段目=履歴を見る/きょうの記録。
- **Home §2**: 犬ごとのホームを1枚のページとして全面スライド(Web: scroll-snap /
  Flutter: PageView全面化+insightsのfamily化)。挨拶・日付・ナビは固定。
  ページ内に名前+ドット+1/N。データは各ページの犬のものだけを描画 (§12)。
- **ヘッダー §3,4**: Dogs/Settingsは「日付(小)+画面タイトル(愛犬/設定)」。
  犬の名前をページタイトルにしない。
- **文言 §5,7,8**: 「◯◯の見守りを終了する」「◯◯のプロフィールを削除する」
  (空名は「この愛犬」)、「新しい愛犬を追加」「見守る愛犬」
  「見守る愛犬を変更します」。
- **写真 §6**: 肉球アイコンタップ→撮る/選ぶ/(登録済みなら)削除/キャンセル。
  中央寄せ正方形トリミング(Web: canvas→dataURL(localStorage) /
  Flutter: image_picker camera+gallery→Firebase Storage)。犬ごとに保持し、
  リング・カード・確認シートに反映。未登録は肉球アイコン。
- **選択UX §9,10**: 減数時の対象選択はラジオ式単一選択(再タップで解除しない)。
  「見守りを終了して変更」→最終確認(見守り終了と同文言)→確認後にのみ適用。
  キャンセルは無変更。
- **§11**: ＋は確認なしで即時反映+「◯頭まで見守れるようになりました」通知。

# v2.3 追補 — 1日1件の健康日誌と日付カード履歴 (両実装済み)

- **きょうの記録 (§2-5)**: 6カテゴリ(散歩/食欲/排便/薬/体調/メモ)を1つのシートで
  入力し「きょうの記録を保存」で一括保存。カテゴリ切替でも入力保持。
  チップは未入力/入力済み(点)/選択中の3状態。当日分は編集として開く。
  未保存で閉じる時のみ確認。選択肢はschema2の`choice`+補足メモ`memo`で構造化。
- **1日1件 (§4,19)**: 保存はカテゴリ別レコードのままupsert(同日同カテゴリは更新)。
  旧schema1(rating)データは削除せず読み取り互換で表示、同日重複は最新を採用。
- **履歴 (§6-18)**: 記録が存在する最新3日分の日付カード+「過去の記録を見る」。
  カード構成=状態→測定(主表示: 状態語→ppm→時刻→回数)→健康日誌→メモ→控えめ導線。
  同日複数測定は同カード内(2件まで一覧、3件以上は「すべて見る」)。
  状態表示は注意結果を優先(要確認>軽い変化>安定>判定不能)しつつ
  ppm/時刻は最新を表示、不一致時は短い補足。文言は §11 の穏やかな4種のみ。
  削除は「…」メニュー(健康日誌のみ削除、測定は残す)。測定なし/日誌なしの日も表示。
