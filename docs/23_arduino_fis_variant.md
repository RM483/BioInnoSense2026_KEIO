# ㉓ 別案: 半導体式(FIS SB-19)+ Arduino Uno R4 WiFi 変種

作成日: 2026-07-21 / ブランチ: `feature/arduino-fis-variant` / 実装: `arduino_fis/`

## 1. 背景・動機

本家は電気化学式(DGS2)+STM32/Leafony。別案として **半導体式(MOS)水素センサ FIS SB-19-00**
を **Arduino Uno R4 WiFi** で読む構成を並行実装する。狙いは「低コスト・小型・入手性」と、
中間発表で設計した**アナログ読み出し回路**をそのまま活かすこと。

半導体式は安価・高感度だが **選択性が低く(VOC/湿度の影響)・ヒータ電力が大きい・ドリフトする**。
そこで本家で作った **BAP(呼気解析)思想が一層効く**：ノイジーで選択性の低い信号を
「意味(呼気イベントの品質付き特徴量)」へ変換する。これ自体が研究的な主張になる。

## 2. 系全体

```
犬の呼気 → SB-19(Rs) → 分圧(RL) → MCP6022 電圧フォロワ → RC LPF
        → R4 14bit ADC → Rs算出(ratiometric) → BAP-lite → HPP → BLE(NUS) → アプリ/クラウド
        ヒータ VH=0.9V ← LT3080(常時)
```
知能を R4 に置き、アプリは「意味」を受け取るだけ、という本家の構図は不変。

## 3. 信号の意味づけ

- `VS = VC·RL/(Rs+RL)`、ADC基準=VC で `Rs = RL·(1/ratio−1)`（電源変動を比率で相殺）。
- `response r = R0/Rs`（清浄大気で≈1、H2上昇でRs減→r増）を呼気信号として扱う。
- BAP-lite: EMA平滑(innovation-gated) → quiet窓でR0学習+ドリフト補償 → onset/offset →
  peak/AUC/rise/duration → **Q(この測定)×C(この計測器) の二軸減点採点**。

## 4. アプリ連携（一貫性の担保）

センサ・MCUが変わっても、**HPPプロトコル(CRC16-CCITT)とBLE(NUS)を本家と共通**にすることで、
既存 Flutter/Web アプリが**ほぼ無改修で接続**できる。変更点は BLE UUID(NUS)と広告名(`Fuwan-R4`)
に合わせた定数のみ（本ブランチで反映済み）。EVT_DATA/RESULT の ppb系は相対H2指標。

## 5. 検証

- `arduino_fis/test`：HPP CRC が CCITT-FALSE(本家互換)、合成MOS呼気カーブで
  READY→ONSET→OFFSET と妥当な Q/C/peak を確認（ホスト、Arduino不要）。
- Web アプリ：tsc/build 緑。

## 6. 本家との比較（要点）

| 観点 | 電気化学式(DGS2) | 半導体式(SB-19) |
|---|---|---|
| H2選択性 | 高 | 中(VOC/湿度に感応) |
| 消費電力 | µA〜mA / コイン電池可 | ヒータ120mW常時 / 電池不向き |
| 出力 | UARTデジタル・ppm近似 | アナログ・相対指標 |
| コスト/サイズ/入手 | 高/大/△ | 低/小/◎ |
| 共通資産 | HPP・BLE・BAP・アプリUI | 同左を流用 |

## 7. 残課題・今後

1. **選択性**: エタノール/VOC/CH4 と H2 の弁別（ヒータ温度変調・複数特徴量・機械学習）。
2. **湿度補償**: 呼気は高湿。RH/温度センサ併設と補償モデル。
3. **較正**: 感度曲線αで相対指標→ppm換算。個体差・経時ドリフトの管理。
4. **電力**: パルス加熱でのデューティ低減（精度とのトレードオフ）。
5. **BAP-lite しきい値**の実犬較正（config.h の BAP_*）。
