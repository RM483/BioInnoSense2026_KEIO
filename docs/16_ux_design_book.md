# 第1章 競合・参照アプリ分析

分析対象は色ではなく **UX・情報設計・遷移・余白・タイポ・操作感**。

## 1.1 健康系 (人間)

| アプリ | 借りるべき設計思想 | 借りないもの |
|---|---|---|
| **Apple Health** | 数値には必ず文脈を添える(「正常範囲内」)。サマリーは編集された少数のカードで、ダッシュボードではない。詳細は常に1タップ奥。階層は色でなく文字サイズで作る | 機能の多さ、リスト密度 |
| **Apple Fitness** | **1画面1モチーフ**(リング)が今日のすべてを語る。他の情報はそのモチーフに従属。達成をイベント(祝福)として演出 | 運動の競争性 |
| **Oura** | アプリが**言葉で話しかける**(「今日は回復に充てましょう」)。スコアより先に文章。グラフには常に正常帯。コーチの静かな口調 | スコア中心主義(数値が主役化しやすい) |
| **Withings** | 医療的信頼は装飾でなく**臨床的な静けさ**(白・精密なタイポ・整った行)から生まれる。測定を時系列のイベントとして扱う。状態チップ(Normal)を数値の隣に | 機器管理UIの事務感 |
| **Gentler Streak** | **許す言葉**(「休んでも大丈夫」)。1画面1決断。感情に寄り添う文言が機能そのもの | イラストの多用(本作は写真=実在の犬が主役) |
| **WHOOP** | (反面教師) データ密度はアスリート向け。飼い主は毎日30秒の安心が欲しいのであって分析ではない | ほぼ全て |

## 1.2 ペット系

| アプリ | 借りるべき設計思想 | 借りないもの |
|---|---|---|
| **Fi** | 犬の写真と名前が常に主役。ハードウェアの存在をUIで消す | ランキング等のゲーミフィケーション |
| **Tractive** | 位置・活動を「いまどうしてる?」の言葉で要約 | 地図中心の密度 |
| **PetPace** | バイタルを正常レンジ帯と併置(獣医向けの正しさ)。我々のゾーン設計の妥当性を裏付け | 臨床表・多段テーブル |

## 1.3 ツール系 (操作感・タイポの規範)

| アプリ | 借りるべき設計思想 |
|---|---|
| **Notion** | UIの消失。コンテンツ(言葉)がインターフェース。余白と行間で階層を作る |
| **Linear** | 8pxリズムの厳密さ。150–250msの気づかないトランジション。色は状態にだけ使う |
| **Arc** | マクロは大胆・ミクロは静か。ブランドの温度は「気配」で出す |

## 1.4 統合原則 (HydroPawの設計原理)

1. **1画面1モチーフ・1決断** (Fitness/Gentler)
2. **言葉が先、数値は後** (Oura) — 数値は詳細画面の住人
3. **正常が主役、警告は必要時だけ** (PetPace/Health)
4. **犬の実在が中心** (Fi) — 写真・名前・「〜ちゃんは元気」
5. **臨床的な静けさ + 優しい言葉** (Withings × Gentler)
6. **UIは消える** (Notion/Linear) — 枠・影・色を足すほど管理画面に近づく

# 第2章 コンセプト3案

ペルソナ: 犬の飼い主(非医療者)。毎日30秒の見守り + 異変時の深掘り +
受診時の証拠。感情ジョブ =「今日も元気?」への即答と安心。

## 案A 「今日の一文」 — Narrative Canvas (Oura × Gentler)

```
┌────────────────────────┐
│ おはようございます            │  ← 挨拶(時間帯)
│                              │
│  ポチは今日も                │  ← 画面の主役は
│  落ち着いています。           │     大きな一文(28pt)
│                              │
│  前回より少し下がって、       │  ← 副文(15pt)
│  いつも通りの範囲です。       │
│                              │
│  ⌄ ここ7日のようす(小さな線)  │
│                              │
│ [ 測定をはじめる ]            │
├──────タブバー──────┤
```
- カードを使わずキャンバスに直接タイポ。最も感情的で最も非管理画面。
- 弱み: 犬の写真の居場所が弱い/情報の足場(トレンド・記録)が遠い/
  文章生成の失敗が即UXの失敗になる。

## 案B 「見守りリング」 — Companion Ring (Fitness × Oura × Fi)

```
┌────────────────────────┐
│ 7月18日(土) おはようございます │
│                              │
│        ╭──────╮            │  ← 状態色のリングが
│       │  🐕写真  │           │     犬のアバターを
│        ╰──────╯            │     やさしく囲む
│      今日は安定しています      │  ← 状態の一文(22pt)
│   通常どおり過ごして大丈夫     │  ← 行動(15pt)
│      最終測定 · 3時間前       │
│                              │
│  ここ7日のようす ▁▂▁▃▂▁▁(線) │
│                              │
│ [ 測定をはじめる ]            │
├──────タブバー──────┤
```
- **リングは進捗やスコアではなく「見守りの輪」**。状態色(緑/琥珀/赤)を
  1箇所だけに宿す。
- 測定体験と造形が貫通する: ホームのリング →(Hero)→ 測定中は呼吸する
  リング → 解析中は静止 → 結果はリングにチェック → ホームへ戻ると
  リングが新しい状態色に変わっている。**アプリ全体が1つのモチーフで語れる**。
- 空状態でもアバター+「はじめての測定をしてみましょう」で成立。
- 弱み: リング=スコアと誤読される懸念 → 数値・%・目盛りを一切描かない
  ことで回避。

## 案C 「日誌」 — Care Journal (Withings × Health)

```
┌────────────────────────┐
│ ポチの見守り日誌              │
│ ── 今日 ──────────────  │
│ ● 安定しています  08:12      │
│ ── きのう ────────────   │
│ ● 安定  21:03 / ● やや高め 08:30│
│ …                            │
│ [ 測定をはじめる ]            │
├──────タブバー──────┤
```
- 記録の連なりが安心を作る。受診時の証拠性が最強。
- 弱み: 「今日大丈夫?」への即答性が先頭エントリ頼み。初日はほぼ空。
  ホームと履歴の役割が重複。

## 2.4 選定: **案B「見守りリング」** (A・Cの長所を吸収)

| 判断基準 | A | **B** | C |
|---|---|---|---|
| 3秒で「今日も元気?」に答える | ○ | **◎ (中央1モチーフ)** | △ |
| 測定イベントとの体験連続性 | △ | **◎ (リングが変形して貫通)** | △ |
| 犬の実在感 | △ | **◎ (アバターが常に中心)** | ○ |
| 空状態の成立 | △ | **○** | × |
| 誤読リスク | 文章依存 | リング=スコア誤読(目盛り排除で回避) | 低 |

吸収: 案Aの「話しかける一文」の語調をリング下の文言に、案Cの日誌構造を
履歴タブ(日付グループ)に採用する。

# 第3章 情報設計・画面遷移

## 3.1 画面構成 (5タブ + フルスクリーンイベント)

```
                    ┌─ フルスクリーン測定イベント ─┐
  Splash → Login →  │ 測定中(呼吸リング)            │
       │            │   ↓ 終了                    │
       ▼            │ 解析中(リング静止+一文)       │
┌─────────────┐     │   ↓ (最低1.2s)              │
│ 🏠 ホーム      │──→│ 結果(リング満色+✓+一文)      │
│ 🩺 測定(準備)  │──→└──────────┬──────────────┘
│ 📈 履歴(日誌)  │                └→ ホームへ(リングが新しい色)
│ 🐶 愛犬       │      履歴行 → 詳細(ppm/波形/統計はここが住所)
│ ⚙️ 設定       │      設定 → デバイス接続
└─────────────┘
```

## 3.2 情報の住所録 (どの情報がどこに住むか)

| 情報 | ホーム | 測定中 | 結果 | 履歴 | 詳細 | 設定 |
|---|---|---|---|---|---|---|
| 状態の言葉 | **主役** | 主役 | 主役 | 行頭 | 見出し | — |
| 行動の一文 | ○ | — | ○ | — | — | — |
| ppm | — | 小さな添字 | 控えめ統計 | 三次色 | **主役** | — |
| 温湿度 | — | — | — | — | ○ | ○(現在値) |
| グラフ | 無数値ミニ線 | 1本の線 | — | ゾーン付き | 波形 | — |
| 電池/接続/BLE | — | 異常時アイコンのみ | — | — | — | **主役** |

## 3.3 ワイヤーフレーム (確定版)

### ホーム
```
おはようございます · 7月18日(土)      ← caption, tertiary
        ╭────────╮
       │ 🐕 avatar │  ← StatusRing 176pt
        ╰────────╯     リング=状態色/1.75pt
   今日は安定しています              ← title 22/700
   前回より改善しています            ← caption 13/600 secondary
   通常どおり過ごして大丈夫そうです   ← body 15/500 primary
   最終測定 · たった今               ← caption tertiary
   ─────────────────
   ここ7日のようす        ✓ 安定    ← overline + chip
   (正常帯つき無数値ライン)
   「正常範囲内で推移しています」
   ─────────────────
   [ 測定をはじめる ]                ← CTA 56pt
```

### 測定イベント (リングの一生)
```
準備:   リング(アバター) + 「ポチの呼気を測定」 + はじめる
測定中: リングが4秒周期で呼吸。中央=状態語、下に小さくppm。
        経過時間は上部に小さく。終了ボタンのみ。
解析中: 呼吸が止まり「解析しています…」— 最低1.2秒の間
結果:   リングが状態色で満ち、✓。「測定できました」+ 状態の一文
        + 控えめ統計(時間/平均/最大) + ホームに戻る
帰還:   ホームのリングが新しい状態色 (AnimatedContainer 400ms)
```

### 履歴 (日誌)
```
履歴
[ゾーン付きトレンド + 要約文]
── 今日 ──
● 安定しています     08:12   6.2ppm ›
── 7月17日(金) ──
● 安定              21:03   5.8ppm ›
```

# 第5章 デザインシステム改訂 (docs/12への差分)

1. **背景の温度**: light背景を `#F2F2F7`(iOS標準灰) → **`#F6F5F2`(暖かい紙色)**。
   医療の冷たさを消し「見守り」の温度に。カードは白のまま(清潔さ=信頼)。
2. **StatusRing** をコアモチーフとして追加(上表)。状態色が現れる場所を
   「リング・状態ドット・チップ・最新点」の4箇所に限定。
3. 挨拶行(おはよう/こんにちは/こんばんは)をホームの1行目に。時間帯で切替。
4. その他(タイポ・8pxグリッド・モーション)は docs/12 を維持。

# 第6章 セルフレビュー (「Appleならここを指摘する」)

実装後に実施し、指摘→修正を記録する。

| # | 想定指摘 | 対応 |
|---|---|---|
| R1 | リングが進捗リングに見える | 目盛り・%・端点を描かない。線幅1.75ptで「輪郭」として描く | 
| R2 | ホームに要素が多い(挨拶+リング+3行+トレンド+CTA) | トレンドはカード化せず平置き・見出し最小化。1スクロール以内 |
| R3 | 状態色の乱用 | 4箇所限定ルール(上記)をコードレビューで担保 |
| R4 | 空状態の弱さ | アバター(肉球)+「はじめての測定をしてみましょう」で初日から成立 |
| R5 | 文章のトーン不一致 | 全文言を「〜しましょう/〜そうです」の非命令形で統一(l10n一元管理) |
