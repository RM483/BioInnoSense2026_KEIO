# HydroPaw Breath Analysis Pipeline (BAP) — アルゴリズム設計書 v1.0

対象: STM32L452RE (Leafony AP03) + DGS2 H2センサ + AC02 BLE
方針: **すべてHAL非依存の純粋C整数演算**で実装し、ホストPCでゴールデン
ベクタ検証してから実機へ載せる。数値は実測(MEASURED)と推定(ESTIMATED)を
必ず区別する(docs/14の規約に従う)。

---

## 0. 研究としての位置づけ — なぜ「取得→送信」では弱いのか

従来構成(改善前)は `DGS2 CSV → レンジ検証 → BLE転送` であり、
知能はすべてアプリ側にあった。これには3つの問題がある。

1. **BLEが切れると知能も消える** — エッジで完結しない。
2. **1Hz生値の垂れ流し**は電力・帯域の無駄(9600bpsのUART層も占有)。
3. **「測定の質」を誰も評価しない** — 犬が呼気を出していなくても
   数値は出る。研究データとして信頼性が担保できない。

BAPは「**呼気イベントをセンサノード上で検出・切り出し・採点する**」
エッジ処理系である。ppmではなく「1回の呼気の特徴量+品質+信頼度」を
一次成果物とする。これがHydroPawの研究的コアである。

---

## 1. パイプライン全景

```
DGS2 1Hz CSV
  │ parse + 物理レンジ検証(既存 dgs2.c)
  ▼
[S1] Hampelフィルタ (median/MAD, w=5)        … スパイク除去
  ▼
[S2] 適応EMA (innovation-gated, Q8固定小数)   … 平滑化と応答性の両立
  ▼
[S3] ベースライン学習 (quiet-window EMA)      … ドリフト補償
  ▼ delta = filtered − baseline
[S4] 呼気セグメンテーション (onset/offset)     … イベント切り出し
  ▼
[S5] 特徴量抽出 (peak/plateau/AUC/rise)       … 64bit整数累積
  ▼
[S6] 品質スコア Q (この測定は良かったか)
[S7] 信頼度スコア C (この計測器は健全か)
  ▼
[S8] VALIDATE → 自動再測定判定 → EVT_RESULT (ARQで確実に届ける)
```

RH(湿度)・温度チャネルは捨てずに **呼気の独立証拠** として使う(§5)。

## 2. 各段の設計判断(他案比較・計算量・メモリ・電力)

### S1: Hampelフィルタ(移動中央値 + MAD外れ値置換), w=5

- **なぜ**: 電気化学センサのUART値には単発スパイク(静電気・電源変動)が
  乗る。平均系フィルタはスパイクを「なまして混ぜる」が、中央値系は
  **除去**できる。MAD(中央絶対偏差)でスケール不変の閾値になる。
  判定: `|x − med| > k·MAD + floor` で中央値に置換(k=3, floor=雑音下限)。
- **持続性判定(スパイク vs ステップの曖昧性解消)**: 呼気前の静穏窓は
  MADが極小のため、**本物の呼気立上りの先頭サンプルもスパイク判定**
  されてしまう(ホストテストが検出した欠陥)。孤立スパイクは1サンプル、
  本物のステップは同方向に≥2連続で逸脱する — この差を使い、
  「同方向2連続の逸脱は受理し、直前の置換もカウント取消」とした。
  代償として同方向2連続スパイクは通すが、単発スパイクという
  ノイズモデル(静電気等)では許容できる。1サンプル分の判定遅延もない。
- **他案比較**:
  - 固定移動平均: スパイクを平均に混入させる。呼気立上りも鈍らせる。却下。
  - 中央値のみ(w=5): 正常値も常に置換され分解能を落とす。Hampelは
    「異常な時だけ介入」する点が優る。
  - 1次元カルマン: プロセスモデル(呼気の動力学)が未知で共分散設計が
    恣意的になる。float必須で検証も不透明。研究説明性の点で却下。
  - Savitzky-Golay: 係数テーブルとFIR履歴が必要で、1Hz・単調な立上り
    検出には過剰。却下。
- **計算量**: w=5挿入ソート2回(値・偏差) ≤ 20比較+置換 ≈ **<300cycles/サンプル**
  (ESTIMATED)。1Hzなので80MHz時のCPU占有 < 0.0004%。
- **RAM**: int32×5(窓) + int32×5(作業) = **40B**。
- **電力影響**: 実質ゼロ(サンプル毎<4µs処理、残りはWFI/STOP2)。

### S2: 適応EMA (innovation-gated), Q8固定小数

- **なぜ**: 平滑化定数αを固定すると「静穏時のノイズ抑制」と「呼気
  立上りへの追従」が両立しない。**イノベーション(新値と推定の差)が
  大きいときだけαを上げる**2段切替(α=1/8 ⇄ 1/2)で両立する。
  y += α·(x−y) をシフト演算のみで実装(Q8: 値を256倍で保持)。
- **他案比較**:
  - 固定α EMA: 立上り追従を優先すればノイズが残り、逆なら呼気開始
    判定が2〜3秒遅れる。
  - 可変αの連続関数(α∝|innov|): 挙動は近いが除算が入り、
    2値切替との差は1Hzでは検証上有意でない。**説明可能性**で2段を採用。
  - IIRバターワース: 係数がfloat/Q15設計になり、群遅延が呼気タイミング
    検出を歪める。却下。
- **計算量**: 減算1・シフト1・加算1 ≈ **<20cycles**。
- **RAM**: int32×2(y_q8, 前回innov) = **8B**。

### S3: ベースラインドリフト補償 (quiet-window learning)

- **なぜ**: 電気化学センサは周囲H2・温度・経時でゼロ点が漂う。
  呼気の意味があるのは「**ベースラインからの上昇分Δ**」である。
  READY(呼気前)中、**窓レンジ(max−min)が閾値未満 ∧ Δがonset閾値の
  半分未満** のときだけ遅いEMA(α=1/64)で学習し、呼気中は凍結する。
  レンジ条件だけでは不十分 — プラトー(呼気維持相)は「静穏だが高い」
  ため、Δ条件がないとベースラインが呼気に引き上げられる
  (ホストテストが検出した欠陥)。同じゲートでRH基準値と呼気前MADも
  スナップショットする(RHはフィルタ遅延がないぶんH2より先に上がる
  ため、onset時点の値では基準として遅すぎる)。
- **他案比較**:
  - 毎セッション'Z'ゼロ校正: 工場校正を上書きし、クリーンエア保証も
    できない(飼い主の部屋で実行される)。破壊的で却下。'Z'は
    メンテナンスコマンドとして手動実行のみ残す。
  - ハイパスフィルタ: プラトー(維持相)まで減衰させ、AUCが歪む。却下。
  - 直前N点平均をベースラインとする: 呼気前に犬が近づいた時点で
    既に上昇していると汚染される。分散ゲート付き学習はこの汚染を防ぐ。
- **計算量**: 分散はWelford逐次(乗算2) ≈ **<50cycles**。
- **RAM**: baseline_q8, 分散用(mean_q8, m2), 窓カウンタ = **20B**。

### S4: 呼気セグメンテーション (onset / offset)

- **onset**: `Δ ≥ CFG_BAP_ONSET_PPB` が **連続 CFG_BAP_ONSET_N 回**(既定
  500ppb×3s)。ヒステリシス+持続時間の二重条件で誤起動を防ぐ。
- **offset**: 次のいずれか — (a) Δがピーク値の `CFG_BAP_OFFSET_PCT`%
  (既定40%)未満に連続3回、(b) 捕捉窓上限 `CFG_BAP_BREATH_MAX_S`(既定90s)、
  (c) READYへ手動停止。(b)は`truncated`フラグ付き。
- **他案比較**:
  - 微分(傾き)閾値: 1Hz・電気化学の応答時定数(t90が十秒オーダー)では
    傾きが浅く、ノイズ微分は悪名高い。振幅+持続の方が頑健。却下
    (傾きは特徴量としてのみ使用)。
  - 機械学習(1D-CNN等): 教師データが存在しない段階では過剰かつ
    検証不能。将来の発展課題として残す(卒論の「今後の展望」向き)。
- **閾値はすべて app_config.h の PROVISIONAL 定数** — 実犬データで
  較正するまで仮値であることを明記(実機較正手順は docs/13 に追記)。

### S5: 特徴量抽出(呼気1回 → 5特徴量)

| 特徴量 | 定義 | 型 | 意図 |
|---|---|---|---|
| peak | max(Δ) | i32 ppb | 従来の代表値 |
| plateau | Δ上位50%サンプルの平均 | i32 ppb | ピークより外乱に強い代表値 |
| AUC | ΣΔ·1s (台形不要, 1Hz等間隔) | u32 ppb·s | 総排出量に比例する生理指標 |
| rise | Δが10%→90%到達の時間 | u16 0.1s | センサ応答+呼気の勢い |
| duration | onset→offset | u16 0.1s | 品質判定にも使用 |

- 64bit累積(sum)はn≤120サンプルで32bitでも溢れないが、AUCは
  120s×120,000ppb=14.4M — 32bitで安全(u32)。sumは既存同様i64。
- **RAM**: 呼気捕捉バッファ int32×96 = **384B** を静的確保。
  O(1)逐次近似(進行中peakの50%以上を逐次平均)も検討したが、peakが
  確定するのはoffset後であり、進行中peak基準では前半サンプルの
  membership誤りが系統誤差になる。384BはSRAM160KBの0.24%であり、
  **正確さを優先してバッファ保持を採用**(rise timeの10/90%判定も
  確定peakに対して正確に計算できる)。

### S6: 品質スコア Q (0–100, この「測定」は信頼できるか)

**減点方式**(開始100点)。減点方式は各項目の寄与が説明可能で、
卒論の表として提示できる。重み付き加算方式は正規化の恣意性が高い。

| 判定 | 条件(既定) | 減点 |
|---|---|---|
| 外れ値率 | Hampel置換率 >5% | −(超過率×4, 最大30) |
| サンプル欠落 | 期待数比 <95% | −(欠落率×2, 最大15) |
| ベースライン不安定 | 呼気前MAD > 200ppb | −15 |
| **RH裏付けなし** | 呼気中ΔRH < +30(3.0%RH) | −20 |
| 短すぎる呼気 | duration < 10s | −20 |
| 窓上限で打切り | truncated | −10 |
| ウォームアップ未了 | 呼気開始が warmup 内 | −25 |

### S7: 信頼度スコア C (0–100, この「計測器」は健全か)

Qと独立の軸として分離する — 「測定は下手だったが機器は正常」と
「機器がおかしい」をアプリが区別できる。**自己診断の集約**。

| 判定 | 条件 | 減点 |
|---|---|---|
| パース失敗率 | セッション中 >2% | −20 |
| 固着検出 | HPP_FLAG_STUCK 発生 | −40 |
| センサ再試行 | タイムアウト再試行 ≥1回 | −10×回数(最大30) |
| 温度が補償範囲外 | −20〜40℃逸脱サンプルあり | −15 |
| ベースライン異常 | baseline > 5,000ppb(周囲大気として不自然) | −20 |
| HPP CRCエラー | セッション中 >0 | −5 |

### S8: VALIDATE と自動再測定

- Q < `CFG_BAP_RETRY_QUALITY`(既定60) かつ 再試行0回 かつ
  セッション残時間あり → **自動でREADYへ戻し再測定**(最大1回)。
  2回目も低品質なら `remeasure_advised` フラグ付きで結果送信
  (最終判断は飼い主に渡す — 押し付けない。docs/17の思想と一致)。

### S9: 温湿度チャネルの利用(補正ではなく「証拠」)

DGS2は**内部で温度補償済み**のppbを出力する(データシート準拠)。
外から重ねて補正すると二重補正になるため**濃度補正はしない**。
代わりに: 呼気は温かく湿っている → **呼気窓中のΔRH・ΔTを
「生体由来の呼気であることの独立証拠」**としてQに組み込む(S6)。
これはガス濃度と無相関なチャネルを品質保証に転用する設計で、
追加ハードウェアなしの多モーダル検証である(新規性ポイント)。
※ロット校正テーブルによる残差補正はフックだけ用意(既定は恒等)。

## 3. 状態機械 v2 (10状態+異常系)

既存6状態(BOOT/SENSOR_INIT/IDLE/MEASURING/SLEEP/ERROR)を保ち、
**呼気セッション用の6状態を追加**する。既存のSTART_CONT/SINGLEは
「ラボモード」(生データ研究用)としてそのまま残す — 研究機材として
生値も取れることは価値であり、後方互換も守られる。

```
                 CMD_BREATH(0x0A)
 IDLE ──────────────────────────► WARMUP ──(warmup済+baseline安定)──► READY
  ▲                                 │timeout(センサ無応答)             │
  │                                 ▼                                │onset
  │◄──(結果送信済/中止)── REPORT ◄─ VALIDATE ◄─ ANALYZE ◄─ BREATH ◄──┘
  │                        │            │低品質&未再試行     ▲
  │                        │ARQ完了     └──────► READY ──────┘ (自動再測定)
  ▼                        ▼
 SLEEP(STOP2)          (EVT_RESULT)
```

異常系(すべてテストで固定):
- WARMUP/READYでセンサ無応答 → 再試行 → E_SENSOR_TIMEOUT → ERROR
- READYで `CFG_BAP_READY_TIMEOUT_MS`(120s) 呼気なし →
  EVT_ERROR(E_NO_BREATH) + IDLE(電池を守る)
- BREATH中にBLE切断(無通信60s) → セッション破棄せず**継続**し、
  結果はARQキューに積む(再接続後に届く) — 従来の「切断=即中止」より
  データを失わない。ラボモードは従来通り安全停止。
- CMD_STOP: どの呼気状態からも安全にIDLEへ(部分結果は破棄、ACK返送)
- 低電池: 測定は中断しない(既存方針)。結果フレームにフラグ。
- IWDG: 全状態でメインループ毎にリフレッシュ(既存)。STOP2はオプション
  バイトIWDG_STOP凍結確認済みのときのみ(既存ガード)。

## 4. BLE信頼性層 (HPP v1.2)

損失モデル: AC02透過UART+BLE Notifyは**順序保証はあるが切断区間で
消失**する。ストリーミング(EVT_DATA)は消えても次で回復するが、
**EVT_RESULT/SUMMARY/ERRORの消失は測定1回の喪失**を意味する。

→ **選択的ARQ**: クリティカルなタイプだけ再送保証する。
全フレームARQ(TCP的)は1Hzストリームでキューを詰まらせ、
遅延と電力を悪化させるだけ — 損失許容度でQoSを分ける設計。

- 新CMD `HPP_CMD_ACK_EVT (0x09)`: payload=ACKするフレームのSEQ。
- FW側: 深さ4のARQキュー。`CFG_ARQ_TIMEOUT_MS`(1000ms)ごとに再送、
  `CFG_ARQ_MAX_ATTEMPTS`(5回)で断念しdropカウンタ(EVT_STATUSで可視)。
- アプリ側: SEQで重複排除(再送で二重受信し得る)、受信即ACK_EVT。
- RAM: 4×(フレーム55B+管理8B) ≈ **252B** (静的確保、malloc不使用)。
- 電力: 再送は損失時のみ。平常時の追加コストは受信ACK 8B/結果1回。

新イベント:
- `HPP_EVT_RESULT (0x86)` 30B: session_id, Q, C, flags, baseline,
  peak, plateau, AUC, rise, duration, temp中央値, ΔRH, 呼気前MAD
- `HPP_EVT_PHASE (0x87)` 2B: 状態遷移時のみ送信(UXの実況用。
  1Hzポーリング不要 → イベント駆動で帯域・電力を節約)

## 5. メモリ・電力予算 (ESTIMATED — 実機ビルドで確定させる)

| 項目 | 追加RAM | 追加Flash | CPU/サンプル |
|---|---|---|---|
| S1 Hampel+持続性判定 | 44B | ~0.5KB | <300cy |
| S2 適応EMA | 8B | ~0.1KB | <20cy |
| S3 ベースライン+RH/MAD基準 | 24B | ~0.3KB | <60cy |
| S4-S5 呼気捕捉+特徴量 | ~440B (バッファ384B含む) | ~0.9KB | <100cy |
| S6-S8 スコア | (finalize時スタックのみ) | ~0.8KB | 呼気終了時のみ ~2k cy |
| ARQキュー | ~260B | ~0.7KB | 再送時のみ |
| 合計 | **~780B** (SRAM160KBの0.5%) | **~3.3KB** (512KBの0.7%) | duty <0.001% |

(sizeof(bap_t)=500B / ARQスロット×4=272B はホストgcc(64bit)実測。
 Cortex-M4(32bit)ではポインタ幅の差でこれ以下になる。
 Flashは実機ビルドで確定させる=ESTIMATED)

電力: 処理は全て整数・サンプル毎<10µs → 追加消費は事実上ゼロ。
支配項は従来通り DGS2(連続~30mA)とBLE。BAPはむしろ
「呼気窓だけ高レートで送る」ことでBLE送信量を減らす方向に働く。

## 6. 新規性ポイント(教員説明用, 12項目)

1. 呼気を「値」でなく「イベント」としてエッジで切り出す設計
2. Hampel+適応EMAの**全整数(Q8)実装** — FPU/floatなしで検証可能
3. 分散ゲート付きベースライン学習(汚染防止の quiet-window 方式)
4. **RH/温度チャネルを品質の独立証拠に転用**(多モーダル自己検証)
5. 品質Q(測定)と信頼度C(機器)の**二軸分離** — 減点方式で説明可能
6. 低品質時の自動再測定(1回)+それでも低品質なら**押し付けず提示**
7. 損失許容度で分けた**選択的ARQ**(全再送でも無保証でもない中間設計)
8. BLE切断中も呼気セッションを継続しARQで後送 — 切断=喪失、を排除
9. イベント駆動EVT_PHASE(ポーリングレス実況)による帯域・電力削減
10. 全パイプラインがHAL非依存純Cで**ホストでゴールデンベクタ検証**可能
11. ラボモード(生値)と製品モード(イベント)の共存 — 研究機材と製品の両立
12. 閾値をPROVISIONALと明示し実犬較正手順に接続(誠実な較正計画)

## 7. 検証計画

- ホスト: 合成呼気波形(立上り+プラトー+回復+スパイク+ドリフト注入)の
  ゴールデンベクタで S1〜S8 を数値固定。ARQは損失注入テスト。
- 実機: docs/13のP1に「呼気モードE2E」「ARQ損失(距離を離す)」を追加。
- スコア較正: 実犬N回の測定でQ分布を確認し、閾値60を再設定する。

## 9. 6人格レビュー(厳しい指摘のみ)と対応記録

### 組み込みエンジニア
- **F1: onset_ms のアンダーフロー** — `now − (N−1)·1000` は起動直後の
  tick値が小さいと巻き戻る。→ 修正済み(クランプ)。
- **F2: 低電池フラグの取りこぼし** — IDLE中に低電池を跨いだ後の
  セッションでは once-per-crossing 設計のため結果フラグに乗らない。
  → 修正済み(セッション開始時に `low_batt_sent` を引き継ぐ)。
- 指摘(仕様として受理): onset判定中のNサンプルの温度/RHは現在値で代表
  される(温度変化は秒オーダーで小さく、系統誤差は無視できる)。
- **F7: 状態番号のログ対応表が旧6状態のまま** — 新状態6..11を
  docs/13参照時に読み違える。→ 本書§8に対応表を追記(下記)。

### BLEエンジニア
- **F3: SEQ重複排除セットが無制限** — SEQは8bit巡回。256フレーム後に
  再利用されたSEQの**新しい結果を誤って破棄する**。→ 修正済み
  (Dart/TSとも直近16件のFIFOに変更。ARQ再送は5秒以内に完結するため
  16件で十分、かつ巡回汚染は起きない)。
- **F4: ラボモードのEVT_SUMMARYがACKされない** — FW v1.2はサマリも
  信頼配送のため、ACKしないと5回再送+drop計上でSN比が下がる。
  → 修正済み(DartのlistenerでもACK返送)。
- 指摘(設計として受理): ACK_EVT自体は再送しない。ACK喪失時はFWが
  再送→アプリが重複排除+再ACKするので結果整合する。
